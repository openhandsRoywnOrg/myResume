# 文档驱动开发（DDD）规范

## 📋 什么是文档驱动开发？

**文档驱动开发（Documentation-Driven Development，简称 DDD）** 是一种开发方法论，核心原则是：

> **先写文档，再写代码**

### 传统开发流程 ❌
```
需求 → 写代码 → 测试 → 补文档（经常省略）
```

**问题**：
- 代码写了一半才发现需求不清晰
- 实现与预期不符
- 文档缺失或过时
- 返工成本高

### 文档驱动开发流程 ✅
```
需求 → 写文档（设计文档） → Review 文档 → 写代码 → 测试 → 更新文档
```

**优势**：
- ✅ 需求在写代码前就清晰
- ✅ 设计问题提前暴露
- ✅ 文档与代码同步
- ✅ 减少返工
- ✅ AI 更容易理解任务

## 🎯 核心原则

### 原则 1: 文档先行

**在写任何代码之前，必须先写文档**：

```markdown
# 功能设计文档

## 1. 需求描述
- 要解决什么问题
- 用户场景是什么

## 2. 设计方案
- 技术选型
- 架构设计
- 接口设计

## 3. 实现计划
- 任务分解
- 优先级

## 4. 验收标准
- 如何判断完成
```

### 原则 2: 文档即契约

**文档是开发团队（包括 AI）之间的契约**：

- ✅ 文档中定义的功能必须实现
- ✅ 文档中定义的接口必须遵守
- ✅ 文档中定义的测试必须通过
- ❌ 不允许未经文档评审的变更

### 原则 3: 文档与代码同步

**代码变更必须伴随文档更新**：

```
修改代码 → 更新文档
新增功能 → 新增文档
删除功能 → 标记文档废弃
```

### 原则 4: 文档可执行

**文档中的示例必须可运行**：

```python
# ❌ 错误：伪代码，无法验证
def example():
    # ... 省略实现 ...
    pass

# ✅ 正确：完整示例，可运行测试
def example():
    """示例函数"""
    result = process_data(input)
    return result
```

## 📁 文档结构

### 文档目录

```
docs/
├── README.md                    # 文档索引
├── DDD_SPEC.md                  # 本文档（文档驱动开发规范）
│
├── requirements/                # 需求文档
│   ├── feature_001_user_auth.md
│   └── feature_002_topic_mgmt.md
│
├── design/                      # 设计文档
│   ├── architecture.md          # 架构设计
│   ├── api_design.md            # API 设计
│   └── database_design.md       # 数据库设计
│
├── guides/                      # 使用指南
│   ├── quickstart.md
│   ├── tutorial.md
│   └── faq.md
│
└── changelog/                   # 变更日志
    └── CHANGELOG.md
```

### 需求文档模板

**文件**: `docs/requirements/feature_XXX.md`

```markdown
# 功能需求：{功能名称}

## 1. 背景

### 1.1 问题描述
当前系统存在什么问题？

### 1.2 用户场景
用户在什么场景下会使用这个功能？

## 2. 需求

### 2.1 功能需求
- [ ] 功能点 1
- [ ] 功能点 2
- [ ] 功能点 3

### 2.2 非功能需求
- 性能要求
- 安全要求
- 兼容性要求

## 3. 验收标准

### 3.1 功能验收
- [ ] 验收标准 1
- [ ] 验收标准 2

### 3.2 测试要求
- 单元测试覆盖率 > 80%
- 集成测试通过
- 回归测试通过

## 4. 参考资料
- 相关文档链接
- 竞品分析
```

### 设计文档模板

**文件**: `docs/design/design_XXX.md`

```markdown
# 设计文档：{设计主题}

## 1. 概述

### 1.1 目标
这个设计要解决什么问题？

### 1.2 范围
包含什么？不包含什么？

## 2. 架构设计

### 2.1 系统架构图
```
[组件 A] → [组件 B] → [组件 C]
```

### 2.2 组件说明
- 组件 A：职责描述
- 组件 B：职责描述

## 3. 详细设计

### 3.1 数据模型
```python
class User:
    """用户模型"""
    id: int
    username: str
    email: str
```

### 3.2 接口设计
```python
def create_user(username: str, email: str) -> User:
    """创建用户"""
    pass
```

### 3.3 流程设计
```
1. 用户提交表单
2. 验证数据
3. 创建用户
4. 发送确认邮件
```

## 4. 实现计划

### 4.1 任务分解
- [ ] 任务 1：创建数据模型
- [ ] 任务 2：实现 API
- [ ] 任务 3：编写测试

### 4.2 优先级
- P0: 核心功能
- P1: 重要功能
- P2: 优化功能

## 5. 风险与缓解

### 5.1 技术风险
- 风险描述 + 缓解措施

## 6. 参考资料
- 相关设计文档
```

## 🔄 开发流程

### 完整流程

```
1. 需求分析
   ↓
   创建需求文档（docs/requirements/）
   ↓
2. 设计评审
   ↓
   创建设计文档（docs/design/）
   ↓
3. 文档 Review
   ↓
   团队 Review 文档（包括 AI）
   ↓
4. 编写代码
   ↓
   按照设计文档实现
   ↓
5. 编写测试
   ↓
   按照验收标准测试
   ↓
6. 更新文档
   ↓
   更新使用指南、API 文档
   ↓
7. 合并 PR
   ↓
   文档和代码一起合并
```

### Issue 中的文档驱动

**Issue 模板应该包含**：

```markdown
## 📝 文档驱动开发

### 1. 需求文档
- [ ] 已创建需求文档：`docs/requirements/feature_XXX.md`

### 2. 设计文档
- [ ] 已创建设计文档：`docs/design/design_XXX.md`
- [ ] 设计文档已 Review

### 3. 代码实现
- [ ] 按照设计文档实现
- [ ] 代码通过 Review

### 4. 测试
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 覆盖率 > 80%

### 5. 文档更新
- [ ] 更新使用指南
- [ ] 更新 API 文档
- [ ] 更新 CHANGELOG
```

## 🤖 AI 与文档驱动

### AI 如何参与文档驱动开发

#### 1. AI 阅读文档理解需求

```markdown
@openhands-agent 请实现这个功能

**必读文档**：
1. docs/requirements/feature_001.md（需求）
2. docs/design/design_001.md（设计）

**任务**：
按照设计文档实现功能
```

#### 2. AI 帮助编写文档

```markdown
@openhands-agent 请帮我写设计文档

**需求**：
实现用户登录功能

**要求**：
- 使用 design_XXX.md 模板
- 包含架构图
- 包含接口设计
- 包含实现计划
```

#### 3. AI 检查文档与代码一致性

```markdown
@openhands-agent 请检查文档与代码是否一致

**检查项**：
- 代码是否实现了文档中的所有功能
- 接口是否与文档一致
- 测试是否覆盖文档中的验收标准
```

## 🚨 常见问题

### Q1: 文档驱动会不会降低开发速度？

**答**：短期看似乎慢了，但长期看更快：

```
传统开发：
写代码 (1 天) → 发现需求问题 → 返工 (2 天) → 补文档 (0.5 天)
总计：3.5 天 + 返工

文档驱动：
写文档 (0.5 天) → Review(0.5 天) → 写代码 (1 天) → 更新文档 (0.2 天)
总计：2.2 天，无返工
```

### Q2: 小功能也要写文档吗？

**答**：根据复杂度决定：

- **简单功能**（< 4 小时）：简化文档，至少写清楚需求和验收标准
- **中等功能**（4-16 小时）：完整需求文档 + 简化设计文档
- **复杂功能**（> 16 小时）：完整需求文档 + 完整设计文档

### Q3: 文档 Review 太耗时怎么办？

**答**：分层 Review：

- **简单功能**：自我 Review + AI Review
- **中等功能**：同事 Review + AI Review
- **复杂功能**：团队 Review + 专家评审

### Q4: AI 能写文档吗？

**答**：可以，但需要人工 Review：

- ✅ AI 可以写初稿
- ✅ AI 可以检查一致性
- ❌ AI 不能完全替代人工 Review
