# 📋 Issue 驱动开发总计划

## 核心理念

```
每个功能 = 一个 Issue = 一个 PR = 完整测试
```

**规则**：
1. 不直接提交代码到 main 分支
2. 每个功能必须通过 Issue 追踪
3. 每个 Issue 必须有验收标准
4. 每个 PR 必须通过测试
5. AI 生成的代码需要人工 Review

## 📝 如何正确提 Issue

### 步骤 1: 选择模板

访问：https://github.com/openhandsRoywnOrg/myResume/issues/new/choose

选择对应的模板：
- **Feature request** - 新功能
- **Bug report** - 报告问题
- **AI Agent task** - 让 AI 完成

### 步骤 2: 填写标题

**好标题**：
```
[Feature] 创建项目基础目录结构
[Feature] 实现 User 模型
[Bug] API 返回 500 错误
[Refactor] 重构 Topic 服务
```

**坏标题**：
```
更新代码
修复 bug
新功能
```

### 步骤 3: 填写正文

#### Feature Request 模板

```markdown
## 相关的问题

清晰描述为什么要做这个功能

例如：
当前项目是单体结构，所有代码在 app.py 中。
随着代码量增加，会导致难以维护和测试。

## 期望的解决方案

描述要实现什么功能

例如：
创建模块化的项目结构，将代码按职责分离到不同目录。

## 验收标准

列出所有必须满足的条件（用复选框）

- [ ] 创建 backend/app/ 目录
- [ ] 创建 models/, services/, api/ 子目录
- [ ] 每个目录有 __init__.py
- [ ] 代码能正常导入
- [ ] 有文档说明结构

## 优先级

- [ ] 低 (有空再做)
- [ ] 中 (可以排期)
- [x] 高 (影响核心功能)
- [ ] 紧急 (阻碍其他工作)

## 实现提示（可选）

给实现者的建议

例如：
1. 先创建目录结构
2. 添加 __init__.py 文件
3. 测试导入是否正常

## 约束条件

实现时的限制

例如：
- 不修改现有业务逻辑
- 保持向后兼容
- 所有文件有适当的注释
```

#### AI Agent Task 模板

```markdown
## 任务描述

详细描述需要 AI 完成的任务

例如：
请实现 User 数据模型，包含用户名、邮箱、密码哈希、角色等字段。

## 具体要求

- [ ] 必须实现 username, email, password_hash, role 字段
- [ ] 必须有 __repr__ 和 to_dict 方法
- [ ] 必须有单元测试
- [ ] 遵循 AGENTS.md 规范

## 验收标准

- [ ] 模型能正常创建和查询
- [ ] 单元测试覆盖率 > 80%
- [ ] 通过 flake8 和 mypy 检查
- [ ] 代码符合项目规范

## 复杂度

- [ ] 简单 (< 50 行代码)
- [x] 中等 (50-200 行代码)
- [ ] 复杂 (> 200 行代码)

## 实现提示

建议参考 ARCHITECTURE_PLAN.md 中的模型设计
使用 SQLAlchemy ORM
添加类型注解和 docstring

## 约束条件

- 必须使用类型注解
- 必须有完整的 docstring
- 必须编写单元测试
```

### 步骤 4: 添加标签

在 Issue 右侧添加标签：

```
📝 enhancement    - 新功能

🔧 refactor      - 重构
📚 documentation - 文档
🧪 tests         - 测试
```

- **Assignees**: 指派给负责的人（或 AI）

## 🔄 Issue 解决流程

### 完整流程图

```
1. 提 Issue
   ↓
2. 讨论和澄清（如有需要）
   ↓
3. 创建分支 (git checkout -b issue-1-feature)
   ↓
4. 实现代码
   ↓
5. 编写测试
   ↓
6. 运行测试 (pytest)
   ↓
7. 代码检查 (flake8, mypy)
   ↓
8. 提交代码 (git commit)
   ↓
9. 推送分支 (git push)
   ↓
10. 创建 PR
   ↓
11. Code Review
   ↓
12. 合并到 main
   ↓
13. 关闭 Issue
```

## 🎯 现在开始：阶段 1

### 立即要做的

#### Issue #1: 创建项目基础目录结构

**你现在可以**：

1. **访问 Issues 页面**
   https://github.com/openhandsRoywnOrg/myResume/issues

2. **点击 "New issue"**

3. **选择 "Feature request" 模板**

4. **复制以下内容**：

```markdown
## 相关的问题

当前项目是单体结构，所有代码在 ai-ops-interview/app.py 中（84 行）。
随着功能增加，代码量会增长到几百上千行，导致：
- 难以找到特定功能的代码
- 修改一个 bug 可能引入新 bug
- 多人协作时代码冲突频繁
- 无法单独测试某个模块

## 期望的解决方案

创建模块化的项目结构，将代码按职责分离：

```
backend/
├── app/
│   ├── __init__.py
│   ├── models/      # 数据模型（User, Topic, Question...）
│   ├── services/    # 业务逻辑（创建、查询、更新...）
│   ├── api/         # API 路由（处理 HTTP 请求）
│   ├── hooks/       # 钩子（自动验证、审计...）
│   └── utils/       # 工具函数
└── tests/
    ├── unit/        # 单元测试
    └── integration/ # 集成测试
```

## 验收标准

- [ ] 创建 backend/app/ 目录
- [ ] 创建 models/, services/, api/, hooks/, utils/ 子目录
- [ ] 每个目录有 __init__.py 文件
- [ ] 创建 tests/unit/ 和 tests/integration/ 目录
- [ ] 能正常导入模块（例如：from app.models import Topic）
- [ ] 更新 README.md 说明新的目录结构

## 优先级

- [ ] 低 (有空再做)
- [ ] 中 (可以排期)
- [x] 高 (影响后续所有开发)
- [ ] 紧急 (阻碍其他工作)

## 实现提示

1. 使用 mkdir -p 创建目录
2. 每个 __init__.py 可以是空文件
3. 创建简单的测试验证导入
4. 不要修改现有业务逻辑

## 约束条件

- 不修改 ai-ops-interview/app.py 的业务逻辑
- 保持向后兼容
- 所有目录有清晰的命名
```

5. **提交 Issue**

## 💡 常见问题

### Q1: 为什么要这么麻烦？直接写代码不行吗？

**答**：现在代码少（84 行）觉得简单，等代码多了（1000+ 行）就会：
- 找不到代码在哪里
- 改一个地方坏三个地方
- 不敢重构

**Issue 驱动** 强制我们：
- 想清楚再写
- 写测试保证质量
- 有人 Review 避免错误

### Q2: 每个小功能都要提 Issue 吗？

**答**：是的！因为：
- 小功能累积成大功能
- 每个功能都需要测试
- 方便追踪和 Review

**例外**：
- 拼写错误修复
- 文档错别字
- 简单的配置修改

### Q3: AI 生成的代码也要提 Issue 吗？

**答**：要！因为：
- AI 可能理解错需求
- 需要人工 Review
- 需要测试保证质量

**流程**：
1. 提 AI Agent Task Issue
2. AI 生成代码
3. 人工 Review
4. 修改问题
5. 运行测试
6. 合并

### Q4: 一个 Issue 太大怎么办？

**答**：拆分成多个小 Issue！

**例如**："实现用户系统" 太大，拆分成：
- Issue #6: User 模型
- Issue #21: JWT 认证
- Issue #22: 注册登录 API

**拆分原则**：
- 每个 Issue 能在 2 小时内完成
- 每个 Issue 能独立测试
- 每个 Issue 有明确的验收标准
